# Legacy System Analysis & Security Practices 🕵️‍♂️

## Overview
This document analyzes the legacy codebase (`config/_legacy_v1`) to identify security features, architectural patterns, and reusable components that should be preserved or migrated to the new version.

## 🛡️ Security Practices (Legacy)

### 1. Authentication & Authorization
*   **RBAC (Role-Based Access Control):**
    *   **Roles:** `MD` (Managing Director) and `Employee`.
    *   **Implementation:** `AuthContext` checked `user.email` against an Allowlist for MDs.
    *   **Priority:** MD Allowlist had "Absolute Priority" over database roles (to prevent lockout).
    *   **Auth Persistence:** Set to `LOCAL` (keeps users logged in).

### 2. Backend Logic (BFF Pattern)
*   **Backend-for-Frontend:** Heavy logic was moved to Node.js (Render) to avoid exposing business logic in the client.
*   **Security Headers:** Backend enforced CORS and verified Firebase ID Tokens.
*   **Service Account:** Used `firebase-admin` with a Service Account Key (kept in server env vars) for privileged operations (Creating users, sending notifications).

### 3. API Key Management
*   **Legacy Strategy:** Used a build script (`scripts/generate-sw-config.cjs`) to inject API keys into `sw.js` at build time.
*   **Vulnerability:** The source file `src/sw.js` ended up containing hardcoded keys in the repository, which we have now **sanitized**.
*   **New Strategy:** We strictly use `import.meta.env` (Vite standards) and `.env` files, which is more secure and standard.

## 🧱 Reusable Modules & "Good Things" to Inherit

### 1. Push Notifications (FCM)
*   **Architecture:** Uses a specific "Background Sync" + "FCM" hybrid.
*   **Service Worker:** Uses `messaging.onBackgroundMessage` (Critical for properly handling background pushes).
*   **Actionable Notifications:** "Mark Attendance" buttons directly in the notification.
*   **Migration Plan:** We should port the `sw.js` logic (sanitized) and `fcm.js` service to the new `src/lib/notifications` module.

### 2. Excel Reporting (`ExcelJS`)
*   **Feature:** Complex report generation with formatting (Yellow for Sundays, Green for Leaves).
*   **Implementation:** Backend controller `exportController.js`.
*   **Value:** High business value. Should be migrated to a Cloud Function or kept in a Node.js microservice.

### 3. Automated Tasks (Cron)
*   **Schedule:** 10:00 AM and 5:00 PM IST reminders.
*   **Tech:** `node-cron`.
*   **Value:** Essential for user engagement.
